## Слайд 2 — Постановка задачи

**Заголовок:** Постановка задачи

**Содержимое:**
- **Задача:** многоклассовая классификация товаров (223 класса)
- **Входные данные:** текст (name, description, model, type_prefix, vendor) + image_url + url
- **Целевая переменная:** `category_ind` — индекс листовой категории в дереве
- **Структура дерева:** двух-трёхуровневая иерархия, например:
  - `Одежда → Женская одежда → Блузы и рубашки`
  - `Мебель → Кровати и матрасы`
- **Метрика:** Accuracy, F1-macro, F1-weighted

**Речь:**
> Формально — это задача многоклассовой классификации с 223 классами.
> У каждого товара есть несколько текстовых полей: название, описание, модель, тип, производитель,
> плюс ссылка на страницу и ссылка на картинку.
> Категории организованы как дерево с глубиной 2-3 уровня.
> Мы предсказываем листовую категорию, то есть самую детальную.
> В качестве метрик смотрим на Accuracy, F1-macro и F1-weighted.
> F1-macro особенно важна, так как отражает качество на редких классах.

**Содержимое:**
- Всего 207 из 223 классов присутствуют в train (16 классов без примеров)
- Распределение крайне несбалансированное:

| Метрика | Значение |
|---|---|

| Среднее | 538 |

**Топ-3 по объёму:**
1. Красота и уход → Парфюмерия — 31 563
2. Мебель → Офисная мебель — 21 760
3. Мебель → Кровати и матрасы — 9 284

**Речь:**

> лидер — Парфюмерия — имеет 31 тысячу примеров. А у 135 классов из 207 — меньше 50 примеров.
> Медиана — всего 25 примеров на класс. Это классический long tail.
> Это объясняет, почему наш baseline показывает Accuracy 95%, но F1-macro только 33% —
> модель хорошо работает на крупных классах и провала ет на редких.
> В дальнейшем нужно специально бороться с этой проблемой.

## Слайд 6 — Архитектура MLOps-проекта

**Заголовок:** Архитектура проекта

**Содержимое:**
```
MLOPSP/
├── src/data/           ← Python-библиотека (pip install -e .)
│   ├── loader.py       load_train / load_test / load_tree
│   ├── preprocessor.py clean_text / combine_text_features
│   └── features.py     TextFeatureExtractor (TF-IDF)
│
├── configs/
│   └── config.yaml     ← единственный источник всех параметров
│
├── scripts/
│   ├── eda.py          ← анализ данных → reports/
│   └── train.py        ← обучение → MLflow + models/
│
├── requirements.txt    ← все версии зафиксированы
└── mlruns/             ← MLflow: параметры, метрики, артефакты
```

**Принципы:**
- Один конфиг управляет всем — меняешь yaml, результаты воспроизводимы
- `src/data` — отдельная библиотека, независимая от скриптов
- MLflow логирует: параметры, метрики, config.yaml, модель

**Речь:**
> Перейдём к архитектуре. Проект организован по принципам MLOps:
> есть отдельная Python-библиотека `src/data` для работы с данными,
> которая устанавливается командой pip install и может быть переиспользована в любом скрипте.
> Все гиперпараметры вынесены в `config.yaml` — это единственный источник правды.
> Хочешь попробовать другой размер словаря или другой C — меняешь конфиг, перезапускаешь скрипт.
> MLflow фиксирует всё: параметры запуска, метрики, сам конфиг и обученную модель.
> Это означает полную воспроизводимость любого эксперимента.

## Слайд 7 — Архитектура модели (бейзлайн)

**Заголовок:** Модель: TF-IDF + Logistic Regression (Baseline)

**Содержимое:**

```
Входные поля: name + description + model + type_prefix + vendor
         ↓
    clean_text()       lowercase, убираем пунктуацию
         ↓
    Конкатенация       одна строка на товар
         ↓
    TF-IDF Vectorizer  max_features=100k, ngram=(1,2), sublinear_tf=True
         ↓
    Sparse matrix      [111k × 100k]
         ↓
    LogisticRegression C=10, solver=saga
         ↓
    category_ind (0..222)
```

**Почему этот baseline:**
- Быстро обучается (< 5 мин на 111k строк)
- Хорошо работает на коротких текстах
- TF-IDF с bigrams ловит фразы типа "нижнее белье", "кухонная мебель"
- sublinear_tf сглаживает влияние частых токенов

**Речь:**
> Baseline — классика для текстовой классификации: TF-IDF плюс логистическая регрессия.
> Препроцессинг простой: лоуеркейс, убираем пунктуацию, конкатенируем все поля через пробел.
> TF-IDF строит словарь из 100 тысяч наиболее информативных токенов и биграмм.
> Биграммы важны — они позволяют поймать устойчивые словосочетания: "нижнее белье", "мягкая мебель".
> sublinear_tf — это логарифмирование частоты термина, что помогает при короткихт екстах.
> Logistic Regression с saga-солвером хорошо работает на разреженных данных такого масштаба.
> Всё обучается меньше чем за 5 минут на 111 тысячах примеров.

## Слайд 8 — Результаты бейзлайна

**Заголовок:** Результаты: Baseline

**Содержимое:**

| Метрика | Значение |
|---|---|
| **Accuracy** | **95.35%** |
| F1-weighted | 94.04% |
| F1-macro | 32.81% |

**Интерпретация:**
- Accuracy 95% — модель правильно классифицирует 19 из 20 товаров
- F1-weighted 94% — хорошо, взвешен по размеру класса, отражает крупные категории
- F1-macro 33% — слабо, равный вес у всех классов, отражает провал на редких категориях
- Разрыв weighted–macro = **61%** — прямое следствие дисбаланса

**Вывод:**
> Модель отлично выучила топ-20 крупных классов, но не справляется с хвостом.
> Это ожидаемое поведение baseline — задача следующих итераций.

**Речь:**
> Результаты baseline: accuracy 95%, F1-weighted 94% — звучит отлично.
> Но F1-macro — только 33%. Это огромный разрыв в 61 процентный пункт.
> Что это значит? Модель прекрасно выучила Парфюмерию и Офисную мебель,
> потому что у них десятки тысяч примеров. Но если у класса 3–5 примеров —
> модель его просто игнорирует, он никогда не попадает в предсказания.
> Это честный и ожидаемый результат первого baseline. Именно поэтому нам нужно двигаться дальше.

## Слайд 9 — Дальнейшее развитие

**Заголовок:** Что дальше?

**Содержимое:**

**Проблемы, которые нужно решить:**
1. Дисбаланс классов → F1-macro 33%
2. Пустые поля у 44-83% строк
3. Не используется URL (структурированная информация о категории)
4. Не используются изображения

**Следующие шаги:**

| Шаг | Идея | Ожидаемый прирост |
|---|---|---|
| URL-признаки | Извлечение домена и пути из URL | +2–5% macro |
| Class weights | `class_weight='balanced'` в LogReg | +5–10% macro |
| LightGBM | Градиентный бустинг на TF-IDF | +3–7% macro |
| Rubert / Multilingual BERT | Fine-tuning трансформера на русских текстах | +15–25% macro |
| CLIP / Vision Transformer | Признаки по картинке | +5–15% macro |
| Иерархическая классификация | Сначала top-level, потом leaf | Лучше на редких |
| Data Augmentation | Перефразировка для редких классов | +5–10% macro |

**Приоритет:**
```
Сейчас: TF-IDF + LogReg (baseline)
РК2:    URL-фичи + BERT на тексте
РК3:    Мультимодальная модель (текст + картинка)
```

**Речь:**
> Что нужно делать дальше? У нас четыре нерешённые проблемы.
> Первая и главная — дисбаланс. Быстрое решение — добавить class_weight='balanced',
> это займёт одну строчку кода и даст заметный прирост F1-macro.
> Вторая — мы вообще не используем URL. А ведь в URL часто зашита категория прямым текстом.
> Третья — нет признаков из картинок. image_url всегда заполнен —
> это значит, мы можем скачать картинки и прогнать через CLIP или ResNet.
> Четвёртая и самая мощная — переход на языковые модели.
> ruBERT или multilingual-BERT fine-tuned на наших данных
> должен дать принципиальный прирост особенно на коротких и неоднозначных текстах.
> На РК2 планирую добавить URL-фичи и BERT, на РК3 — мультимодальность.
